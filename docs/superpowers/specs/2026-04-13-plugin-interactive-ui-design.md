# Plugin Interactive UI — Design Spec

**Date:** 2026-04-13
**Status:** Approved

## Overview

Plugins can pause mid-execution to present an interactive form to the user inside the processing progress panel. The pipeline timer freezes, the user responds (or skips), and execution resumes with the user's input.

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| When does UI appear? | Mid-execution (plugin pauses) | Plugin needs runtime data to build the form |
| Input types | Arbitrary — text, number, boolean, select, multi-select, textarea | Full flexibility for plugin authors |
| Form definition | HTML/markdown content + typed input fields array | Rich prompts without per-plugin Angular components |
| Where in the UI | Inline in the processing progress panel | Part of the natural pipeline flow |
| Timer behavior | Pauses with "Waiting for input" state | Clear signal the ball is in the user's court |
| Multiple pauses | Yes, a plugin can call `requestInput` multiple times | Supports multi-round interactions |
| Skip behavior | User can submit or skip (defaults applied) | Plugin controls whether skip is available |
| Interaction hint | Static `requiresInteraction` metadata + runtime decision | Users know upfront; plugin has final say |
| Transport | SSE for server→client events, REST POST for client→server responses | Extends existing infrastructure |

## 1. Plugin Metadata Changes

`PluginMeta` gains one optional field:

```ts
export interface PluginMeta {
  // ... existing fields
  /** Hint: this plugin may pause execution to ask the user for input */
  requiresInteraction?: boolean;
}
```

Static hint shown during pipeline configuration. Plugins with `requiresInteraction: true` display an "Interactive" badge in the pipeline configurator. The actual decision to show a form happens at runtime inside `execute()`.

## 2. Form Definition Schema

### InputField

```ts
export interface InputField {
  id: string;
  type: 'text' | 'number' | 'boolean' | 'select' | 'multi-select' | 'textarea';
  label: string;
  description?: string;
  required?: boolean;
  defaultValue?: unknown;
  options?: Array<{ label: string; value: string }>;
  validation?: {
    min?: number;
    max?: number;
    minLength?: number;
    maxLength?: number;
    pattern?: string;
  };
}
```

### InputRequest

```ts
export interface InputRequest {
  requestId: string;          // generated by the server
  pluginId: string;
  title: string;              // form heading
  content?: string;           // HTML/markdown body
  fields: InputField[];       // can be empty for confirmation-only prompts
  skippable: boolean;
  skipLabel?: string;         // e.g. "Use defaults"
  submitLabel?: string;       // e.g. "Confirm speakers"
}
```

### InputResponse

```ts
export interface InputResponse {
  requestId: string;
  skipped: boolean;
  values: Record<string, unknown>; // field id → value, empty if skipped
}
```

## 3. Server-Side Pause/Resume Mechanism

### PipelineContext addition

```ts
export interface PipelineContext {
  // ... existing fields
  requestInput: (request: Omit<InputRequest, 'requestId' | 'pluginId'>) => Promise<InputResponse>;
}
```

### Pipeline service internals

- `pendingInputs: Map<string, { resolve: (r: InputResponse) => void }>` — in-memory store
- When `requestInput()` is called:
  1. Generate a UUID `requestId`
  2. Store a Promise resolver in `pendingInputs`
  3. Broadcast SSE event `plugin:input-requested` with the full `InputRequest`
  4. Return the Promise — the `await plugin.execute(ctx)` naturally suspends
- When the client POSTs to `/api/plugins/input/:requestId`:
  1. Look up the resolver in `pendingInputs`
  2. Call it with the `InputResponse`
  3. Remove from map, return 200

### New REST endpoint

```
POST /api/plugins/input/:requestId
Body: InputResponse
Response: 200 OK
```

### SSE events

- `plugin:input-requested` — contains the full `InputRequest` plus `waitingForInput: true`
- `plugin:input-received` — signals the client to resume the timer

## 4. Client-Side Processing Panel Changes

### New signals on ProcessingProgressComponent

```ts
readonly pendingInput = signal<InputRequest | null>(null);
readonly waitingForInput = signal(false);
```

### SSE event handling

- `plugin:input-requested` → set `pendingInput`, set `waitingForInput(true)`, pause ticker
- `plugin:input-received` → clear `pendingInput`, set `waitingForInput(false)`, resume ticker

### Template behavior

When a step is `running` and `waitingForInput()` is true:
- Spinner replaced with "Waiting for input" badge
- Progress bar hidden
- `PluginInputFormComponent` rendered inline below the step header

### New component: PluginInputFormComponent

- **Location:** `client/src/app/features/onboarding/plugin-input-form/plugin-input-form.component.ts`
- **Input:** `InputRequest`
- **Outputs:** `submitted(InputResponse)`, `skipped(InputResponse)`
- Builds a reactive `FormGroup` dynamically from `fields`
- Renders `content` as sanitized HTML above the form fields
- Renders each field type: text input, number input, checkbox, select dropdown, multi-select, textarea
- Enforces validation from `required` and `validation` rules
- Submit button (primary) and Skip button (secondary, only if `skippable`)
- Custom button labels from `submitLabel`/`skipLabel`

### Parent wiring

`ProcessingProgressComponent` catches the output events and calls `PluginService.submitInput(requestId, response)` which POSTs to `/api/plugins/input/:requestId`.

## 5. Plugin Author Experience

```ts
async execute(ctx: PipelineContext): Promise<PipelineContext> {
  const speakers = detectSpeakers(ctx.clips);

  const response = await ctx.requestInput({
    title: 'Speaker Assignment',
    content: `<p>Found <strong>${speakers.length}</strong> speakers.</p>`,
    fields: speakers.map(s => ({
      id: `speaker_${s.id}`,
      type: 'text' as const,
      label: `Label for Speaker ${s.id}`,
      defaultValue: s.suggestedName,
      required: true,
    })),
    skippable: true,
    skipLabel: 'Use suggested names',
    submitLabel: 'Apply labels',
  });

  if (!response.skipped) {
    speakers.forEach(s => {
      s.name = response.values[`speaker_${s.id}`] as string;
    });
  }

  // Can call requestInput again for subsequent interactions
  return ctx;
}
```

Key points:
- `requestInput` is async — just `await` it
- Check `response.skipped` to decide whether to use defaults
- Can call `requestInput` multiple times in one `execute()`
- `content` supports HTML, sanitized on the client
- Fully backward compatible — plugins that don't call `requestInput` are unaffected

## 6. Error Handling and Edge Cases

| Scenario | Behavior |
|----------|----------|
| **No timeout** | User takes as long as needed. Timer is paused, no resource cost. |
| **Browser refresh** | SSE reconnects. Server re-broadcasts the pending `InputRequest` from the `pendingInputs` map. |
| **Pipeline abort** | Pending Promise rejected with abort error. Plugin's `execute()` throws, caught by existing pipeline error handling. `pipeline:error` broadcast as usual. |
| **Invalid requestId** | POST returns 404. |
| **Concurrent pipelines** | Each `requestId` is a UUID — no collisions. |
| **Content sanitization** | `DomSanitizer.bypassSecurityTrustHtml` — same pattern as `coloredMessage()`. Plugins are trusted server-side code. |
